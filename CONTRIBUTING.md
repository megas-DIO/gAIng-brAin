# Contributing to gAIng-brAin

Thank you for your interest in contributing to the gAIng War Room! This document provides guidelines for contributing to this repository.

## Before You Start

**Required Reading:**
1. Read `/docs/GAING_WAR_ROOM.md` - Understand the War Room concept and workflow
2. Read `/docs/CANON.md` - Know the project standards and constraints

## How to Contribute

### For AI Agents

1. **Only work on assigned Issues**
   - Do not start work without an Issue
   - Wait to be assigned to an Issue
   - Stay within the Mission scope

2. **Follow the workflow**
   - Read the Issue completely
   - Check Canon for standards
   - Create a feature branch
   - Make your changes
   - Submit a Pull Request
   - Link PR to the Issue with "Closes #123"
   - Wait for review
   - Do NOT self-merge

3. **Code Standards**
   - Follow `/docs/CANON.md` coding standards
   - Write clear commit messages
   - Test your changes
   - Update documentation if needed

### For Human Contributors

1. **Create an Issue first**
   - Use the Mission template
   - Describe what you want to do and why
   - Wait for discussion/approval

2. **Fork and branch**
   - Fork this repository
   - Create a feature branch from `main`
   - Use naming: `feature/description` or `fix/description`

3. **Make your changes**
   - Follow the coding standards in `/docs/CANON.md`
   - Write tests for new features
   - Keep changes focused and atomic

4. **Submit a Pull Request**
   - Use the PR template
   - Link to the related Issue
   - Describe your changes clearly
   - Ensure all checks pass

5. **Review process**
   - Respond to feedback
   - Make requested changes
   - Do not merge your own PR
   - Wait for approval from @megas-DIO

## What We're Looking For

✅ **Good Contributions:**
- Bug fixes with clear reproduction steps
- New features discussed in Issues first
- Documentation improvements
- Test coverage improvements
- Code that follows Canon standards

❌ **Avoid:**
- Surprise PRs without Issues
- Breaking changes without discussion
- Code that violates Canon constraints
- Incomplete or untested changes
- Massive refactors

## Pull Request Checklist

Before submitting your PR, ensure:

- [ ] I have read `/docs/CANON.md` and `/docs/GAING_WAR_ROOM.md`
- [ ] My code follows the project's coding standards
- [ ] I have created an Issue and this PR links to it
- [ ] I have tested my changes
- [ ] I have updated documentation if needed
- [ ] My commits have clear messages
- [ ] I have not violated any Hard Constraints in Canon
- [ ] This PR does only what the Issue describes

## Code Review Process

1. **Automatic checks** run first (if configured)
2. **Code owner review** (@megas-DIO) is automatically requested
3. **Feedback** may be provided as comments
4. **Changes** may be requested
5. **Approval** is given when ready
6. **Merge** is performed by Ryan

## Communication

- **Questions about an Issue?** Comment on the Issue
- **Discussion about approach?** Use GitHub Discussions
- **Bug report?** Create an Issue with bug details
- **Feature idea?** Create an Issue with `[IDEA]` prefix

## Hard Rules

 These are non-negotiable:

1. **No work without an Issue** - Every PR must link to an Issue
2. **No self-merging** - Only @megas-DIO merges PRs
3. **No secrets in code** - Never commit API keys, passwords, tokens
4. **No breaking Canon** - Hard Constraints must be respected
5. **No scope creep** - PRs must stay within their Issue scope

## Getting Help

If you're stuck:

1. Check `/docs/CANON.md` for standards
2. Search existing Issues for similar problems
3. Comment on your Issue asking for clarification
4. Use GitHub Discussions for general questions
